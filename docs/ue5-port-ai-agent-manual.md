# 1. 你最终会得到什么

当你全部做完之后，你会得到一套这样的效果：

## 1.1 在大屏上能看到
- 港口场景背景（来自 UE5.7）
- 天气信息
- 台风路径简图
- 航线信息
- 堆场状态
- 告警信息
- AI 智能体聊天面板
- AI 自动播报栏
- 场景控制按钮
- 设备控制按钮

## 1.2 用户可以做什么
- 在大屏里向 AI 直接提问，例如：
  - “现在台风风险大吗？”
  - “需不需要停止岸桥作业？”
  - “当前堆场哪个区域最拥堵？”
- 点击按钮触发：
  - UE5 场景动画
  - 模拟设备控制
- 当后台模拟出“风速升高、台风逼近、设备告警”等事件时：
  - AI 会自动生成播报文案
  - 前端会自动显示播报
  - UE5 未来也可以同时收到同一条播报并做视觉联动

# 2. 先搞清楚整个系统怎么分层

先不要急着写蓝图、写接口、接模型。
你必须先理解：这个系统不是“只有 UE5 一个东西”，而是 4 层。

```text
第 1 层：UE5.7 场景层
- 港口码头三维场景
- 船、吊机、泊位、堆场、天气特效
- UMG 大屏

第 2 层：前端大屏层（本仓库 ui）
- 聊天面板
- 台风图
- 天气卡片
- 告警列表
- AI 播报
- 控制按钮

第 3 层：后端服务层（本仓库 backend）
- 提供数据接口
- 提供 AI 问答接口
- 提供命令接口
- 提供 WebSocket 实时推送
- 模拟天气、台风、港口数据

第 4 层：免费 AI 模型层（Ollama）
- 真正负责生成自然语言回复
- 真正负责生成智能播报文案
```

你要记住一个核心原则：

## 2.1 UE5 负责“看起来像真的”
也就是：
- 场景
- 动画
- 灯光
- 天气特效
- UI 容器

## 2.2 后端负责“逻辑和数据”
也就是：
- 当前风速是多少
- 台风是否接近
- 当前有哪些告警
- AI 当前应该说什么
- 点击按钮之后应该触发什么动作

## 2.3 AI 负责“把数据翻译成人话”
也就是：
- 根据天气、台风、堆场、告警数据生成分析
- 给用户做问答回复
- 自动播报突发事件

# 3. 仓库里每个目录是干什么的

你先要知道仓库里的文件分别干嘛，不然你后面会改乱。

## 3.1 `ui/`
这是前端大屏。

你可以理解为：
- 这不是 UE5 场景本体；
- 这是覆盖在场景上面的一层“业务大屏 UI”。

这里面最重要的文件：

### `ui/index.html`
页面结构。
负责定义：
- 左侧天气、航线、台风、告警区域
- 中间 UE5 占位层和 AI 智能体面板
- 右侧控制中心和播报区

### `ui/styles.css`
页面样式。
负责定义：
- 面板玻璃风格
- 按钮样式
- 聊天区样式
- 播报横幅样式

### `ui/app.js`
页面逻辑。
负责处理：
- 调用后端接口
- 建立 WebSocket
- 接收播报
- 更新天气、台风、告警、KPI
- 给 AI 发送聊天请求

## 3.2 `backend/PortDigitalTwin.Api/`
这是 .NET 8 后端。

它的作用是：
- 给前端和 UE5 提供统一的数据服务
- 调用 Ollama 免费模型
- 定时模拟天气和台风变化
- 生成自动播报

### 里面最重要的文件

#### `Program.cs`
后端入口。
负责注册接口和启动服务。

#### `Services/AiAgentService.cs`
AI 智能体服务。
负责：
- 调用 Ollama
- 如果 Ollama 不可用，就退回规则引擎回复
- 生成播报文案

#### `Services/PortStateStore.cs`
数字孪生状态仓库。
负责保存：
- 天气状态
- KPI
- 台风路径
- 堆场状态
- 告警列表
- 播报列表

#### `Services/BroadcastWorker.cs`
后台定时任务。
负责每隔几秒：
- 更新模拟天气
- 更新模拟台风
- 触发告警
- 生成一条 AI 播报
- 推送给前端/UE5

## 3.3 `docs/`
这是文档目录。

里面有两份你要重点看：
- `docs/ue5-port-ai-agent-solution.md`：总体方案
- `docs/ue5-port-ai-agent-manual.md`：也就是你现在正在看的超详细说明书

# 4. 开始之前你需要准备什么

这一部分非常重要。
如果环境没准备好，后面会一直卡。

## 4.1 你至少需要准备 3 类环境

### A. 用来跑前端的环境
你需要至少有以下任意一个：
- Python 3
- 或者 Node.js
- 或者任意一个本地静态资源服务器

本手册使用的是：
- `python3 -m http.server`

### B. 用来跑后端的环境
你需要：
- .NET 8 SDK

如果没有 .NET 8，后端启动不了。

### C. 用来跑免费 AI 的环境
你需要：
- Ollama

如果没有 Ollama，也没关系：
- 系统仍然能跑；
- 只是 AI 会退回规则引擎；
- 也就是它还是能回答，但不是真正的大模型回答。

# 5. 第一步：先把前端大屏跑起来

这一步的目标非常简单：

> 不管后端和 AI 有没有装好，先让你看到大屏页面能打开。

## 5.1 打开终端
进入仓库目录：

```bash
cd /workspace/UE5.7_SmartPort
```

## 5.2 启动前端静态服务
执行：

```bash
cd ui
python3 -m http.server 8080
```

这条命令是什么意思？
- `cd ui`：进入前端目录
- `python3 -m http.server 8080`：在本地 8080 端口启动一个静态文件服务

## 5.3 打开浏览器
浏览器输入：

```text
http://127.0.0.1:8080
```

或者：

```text
http://localhost:8080
```

## 5.4 你现在应该看到什么
如果正常，你应该能看到：
- 左侧天气卡片
- 航线列表
- 台风路径简图
- 告警列表
- 中间 UE5 占位层
- 中间 AI 智能体聊天区域
- 右侧控制按钮
- 右侧 AI 智能播报区

## 5.5 如果此时后端没启动，会发生什么
这是正常的。

页面仍然能打开，因为前端已经内置了离线 mock 数据。
也就是说：
- 天气能显示
- 台风能显示
- 聊天区能显示
- 但点击“连接后端”后，拿不到真实接口返回

这不是错误，这是设计好的离线演示模式。

# 6. 第二步：再把后端服务跑起来

这一步的目标是：

> 让前端不再只是静态页面，而是真正能从后端拿到动态数字孪生数据。

## 6.1 先确认你机器装了 .NET 8
终端执行：

```bash
dotnet --version
```

### 正常情况
如果返回类似：

```text
8.0.xxx
```

说明可以继续。

### 不正常情况
如果提示命令不存在，说明你没装 .NET 8 SDK。
这时你先不要继续折腾代码，先安装 .NET 8。

## 6.2 进入后端目录
执行：

```bash
cd /workspace/UE5.7_SmartPort/backend/PortDigitalTwin.Api
```

## 6.3 还原依赖
执行：

```bash
dotnet restore
```

这一步是下载并还原项目依赖。

## 6.4 启动后端
执行：

```bash
dotnet run
```

## 6.5 启动成功后会发生什么
正常情况下，后端会在本地启动一个 Web 服务。

默认你应该把它理解为：

```text
http://localhost:5000
```

前端默认也是按这个地址去找后端的。

## 6.6 后端启动后，当前系统已经有哪些接口

### 获取当前数字孪生快照
```text
GET /api/twin/snapshot
```

作用：
- 获取当前天气
- 获取当前 KPI
- 获取当前台风路径
- 获取当前堆场状态
- 获取当前告警
- 获取当前播报

### AI 聊天接口
```text
POST /api/chat
```

作用：
- 把你的问题和当前港口状态一起发给 AI
- AI 返回自然语言回答

### 命令接口
```text
POST /api/commands/execute
```

作用：
- 前端点击按钮后调用这个接口
- 后端返回一个“已接收”的确认
- 未来 UE5 和设备控制也可以用这个接口做统一命令入口

### WebSocket 实时通道
```text
/ws
```

作用：
- 自动推送天气变化
- 自动推送台风变化
- 自动推送 AI 播报
- 自动推送命令确认

# 7. 第三步：接入免费 AI 大模型 Ollama

这一步的目标是：

> 把“规则回复”升级成真正的大模型回复。

## 7.1 什么是 Ollama
你可以把 Ollama 理解成：
- 一个本地大模型运行器
- 类似你自己的本地 AI 服务端
- 不需要每次都调用外部收费 API

## 7.2 为什么这里推荐 Ollama
因为它满足你当前项目最重要的几个需求：
- 免费
- 本地可跑
- 演示方便
- 中文模型可选
- 对接简单

## 7.3 安装完成后先启动 Ollama 服务
执行：

```bash
ollama serve
```

这条命令的意思是：
- 启动一个本地 AI 服务
- 默认监听 `http://127.0.0.1:11434`

## 7.4 拉取推荐模型
执行：

```bash
ollama pull qwen2.5:3b-instruct
```

为什么推荐这个模型？
- 中文表现不错
- 机器要求相对低
- 适合你这种“数字孪生问答/播报”场景

## 7.5 后端为什么会自动识别它
因为后端配置文件里已经写好了默认配置：
- Provider = Ollama
- BaseUrl = `http://127.0.0.1:11434`
- Model = `qwen2.5:3b-instruct`

所以只要你把 Ollama 服务跑起来，后端就会优先调用真正的大模型。

## 7.6 如果 Ollama 没装，会不会整个系统报废
不会。

系统会自动退回“规则引擎回答”，也就是：
- 问天气，按当前天气字段拼接回答
- 问堆场，按当前堆场字段拼接回答
- 问台风，按当前告警和风速拼接回答

这意味着：
- 你的演示不会中断；
- 只是回答没那么自然。

# 8. 第四步：验证前端、后端、AI 是否已经打通

这一部分很重要。
你不能只看“程序启动了”，你要真的验证链路。

## 8.1 验证前端页面是否正常
浏览器打开：

```text
http://127.0.0.1:8080
```

如果能打开页面，说明前端静态服务正常。

## 8.2 验证后端快照接口是否正常
浏览器或接口工具打开：

```text
http://127.0.0.1:5000/api/twin/snapshot
```

如果返回一段 JSON，说明后端成功。

你应该能看到类似字段：
- `portName`
- `kpi`
- `weather`
- `routes`
- `typhoonTrack`
- `yard`
- `alarms`
- `broadcasts`

## 8.3 验证前端是否成功拿到后端快照
回到前端页面，点击右上角：

```text
连接后端
```

如果成功：
- 连接状态会变成“已连接”
- 日志里会提示连接成功
- 页面数据会开始跟随后台更新

## 8.4 验证 AI 是否已经可用
在聊天框里输入：

```text
当前台风风险如何？
```

然后点击发送。

### 如果 Ollama 正常
你会得到更自然、更像真实助手的回答。

### 如果 Ollama 没正常
你仍然会得到回答，只是它来自 fallback 规则引擎。

## 8.5 验证自动播报是否正常
等几秒钟。
因为后台有个定时任务会自动更新模拟天气/台风数据。

如果正常：
- 页面播报栏会更新
- 中间 UE5 占位层底部的播报 banner 会变化
- 聊天区里也会追加一条 AI 自动播报

## 8.6 验证按钮命令是否正常
你可以试着点：
- 播放吊机动画
- 船舶进港演示
- 台风预警演示
- 闸机开启
- 紧急停止

当前版本里，这些命令会：
- 发给后端
- 后端生成确认信息
- 前端收到确认后写入日志

这一步虽然还没真实控制设备，但已经把“控制链路”打通了。

# 9. 第五步：把这个系统接进你的 UE5.7 项目

这一步是最关键的。
也是最容易做错的。

你现在已经有：
- UE5.7 港口场景
- UMG 封面大屏

你要做的，不是把所有逻辑都塞回 UMG，
而是要先决定你想走哪条路。

## 9.1 路线 A：UE5 场景 + Web 大屏覆盖（推荐）

这是目前最稳、最快、最容易演示成功的路线。

### 这条路线怎么理解
- UE5 继续负责三维场景
- 这个仓库里的 `ui/` 负责大屏 UI
- 你把 Web UI 叠加在 UE5 画面上面

### 适合你当前情况的原因
因为你已经有场景，但没有数据系统。
这种时候：
- 让 UE5 专注场景；
- 让 Web 专注数据面板和 AI；
是成本最低的方式。

## 9.2 具体操作方式 1：在 UMG 里嵌入 Web Browser Widget

### 步骤 1：打开你的 UE5 工程
打开你当前的 UE5.7 项目。

### 步骤 2：找到你的大屏 UMG Widget
也就是你现在这张封面 UI。

### 步骤 3：在 UMG 中加入一个 Web Browser Widget
把它理解成：
- 在 UMG 里放一个浏览器窗口
- 这个浏览器窗口加载我们本地的前端大屏页面

### 步骤 4：给这个浏览器控件设置地址
地址填：

```text
http://127.0.0.1:8080
```

如果前端部署在别的机器，就改成对应 IP。

### 步骤 5：调整布局
你可以把这个 Web Browser 放在：
- 右侧信息栏位置
- 底部 AI 值班席位置
- 或者整张屏幕作为覆盖层

### 步骤 6：运行 UE5 进行测试
如果正常：
- 你的 UMG 界面里会直接显示这套 AI 大屏
- 里面的聊天、播报、天气、台风都会跟着后端更新

## 9.3 具体操作方式 2：用 Pixel Streaming / 视频流做背景

如果你是想让：
- UE5 全屏做背景
- Web UI 完全盖在上层

那么流程通常是：
- UE5 通过 Pixel Streaming 输出画面
- Web 页面把 UE5 画面当背景层
- UI 组件压在最上层

这种方式更像真正的展示大屏。
但它比直接 UMG 嵌网页稍微复杂一点。
如果你当前的目标是“尽快成型可演示”，建议先走 **UMG 嵌 Web**。

# 10. 第六步：如果你想继续用 UMG 原生控件，而不是网页嵌入，应该怎么做

有些项目不希望嵌网页，而是希望：
- 原来的 UMG 大屏继续保留；
- 只把数据和 AI 能力接入进来。

这是可以的。

你要做的事情分成 3 类。

## 10.1 在 UE5 中定时请求快照接口
你需要让 UE5 定时请求：

```text
GET /api/twin/snapshot
```

作用是：
- 读取天气
- 读取 KPI
- 读取台风路径
- 读取告警
- 读取播报

然后把这些数据显示到：
- TextBlock
- RichTextBlock
- ListView
- 自定义图表控件

## 10.2 当用户在 UMG 中提问时，调用 AI 接口
你需要从 UMG 收集用户输入文本，然后发送到：

```text
POST /api/chat
```

请求体的思路是：
- message：用户问题
- sessionId：会话 ID
- includeSnapshot：是否附带快照

后端返回后，你把 AI 的回答显示到 UMG 文本区域即可。

## 10.3 建立 WebSocket 连接接收实时播报
你还要让 UE5 连接：

```text
/ws
```

当收到推送时：
- 如果是天气变化，就更新天气面板
- 如果是播报消息，就更新播报区
- 如果是告警，就让界面闪烁或者弹窗
- 如果是命令确认，就更新设备状态或动画状态

# 11. 第七步：如何把播报联动到 UE5 的场景效果

这一部分，是真正让“数字孪生”看起来高级的关键。

你的目标不是只让页面出现一段字，
而是要让这段字和场景一起变化。

## 11.1 你应该建立一个播报等级规则
比如：

### `info`
普通信息播报。
场景只需要轻微提示。

你可以做：
- 底部字幕条显示
- 指示灯轻微闪动

### `warning`
预警级播报。
场景需要明显提示。

你可以做：
- 播放风声
- 打开黄色警示灯
- 将桥吊、泊位区域做高亮
- 播放提示音

### `critical`
严重级播报。
场景需要强烈联动。

你可以做：
- 红色警报灯
- 大风雨特效增强
- 相机切到关键区域
- UI 弹出强提醒
- 禁用部分控制按钮

## 11.2 一个典型联动例子：台风预警
假设 WebSocket 收到：
- 风速达到 25m/s
- 告警等级 = critical

那么你在 UE5 可以这样联动：

### 第一步：切换天气状态
- 云层变暗
- 雨粒子加强
- 海面材质波浪增强

### 第二步：切换灯光与警示
- 泊位周围红色警示灯开启
- UI 显示“台风预警”

### 第三步：设备演示联动
- 桥吊停止动画
- 船舶减速
- 某些高风险设备按钮不可点击

### 第四步：播放字幕播报
把后端推送的 `broadcast.content` 直接显示在：
- UMG 底部横幅
- 或者大屏中央弹窗

# 12. 第八步：如何替换模拟数据为真实数据

当前仓库里，后端默认是模拟数据。
这样做的目的，是为了让你 **不用等真实接口、也能先把系统架起来**。

等你演示通了，再替换真实数据。

## 12.1 你需要替换哪些模拟数据
主要有 4 类：

### A. 天气数据
当前：后端随机模拟风速、降雨、能见度。

未来你可以替换成：
- 第三方天气 API
- 你们港口自己的气象站接口
- 风速仪/雨量计上传数据

### B. 台风数据
当前：后端按内置规则移动台风点位。

未来你可以替换成：
- 台风路径接口
- 业务系统推送
- 自己的预警平台数据

### C. 港口业务数据
当前：
- 航线
- KPI
- 堆场状态

都是模拟值。

未来你可以替换成：
- TOS 系统
- WMS 系统
- 调度系统
- 泊位系统

### D. 设备状态
当前：
- 命令只做模拟确认
- 没有真实硬件执行

未来你可以替换成：
- MQTT
- OPC UA
- Modbus TCP
- 设备厂商 SDK

## 12.2 替换时最重要的原则
不要让前端和 UE5 直接对接各种数据源。

正确做法是：
- 真实数据先进入后端
- 后端统一整理成 `TwinSnapshot`
- 前端和 UE5 只拿统一快照

这样做的好处是：
- 前端不会乱
- UE5 不会乱
- AI 也有统一的数据输入

## 13.1 第一个阶段：先让“AI 会说话”

# 14. 常见问题排查

## 14.1 页面能打开，但是点“连接后端”没反应
先检查：
- 后端有没有启动
- 后端是不是跑在 `localhost:5000`
- 前端 `ui/app.js` 里的地址是不是和后端一致

## 14.2 聊天可以发，但回答很机械
这通常说明：
- 后端是正常的
- 但是 Ollama 没有真正连上
- 所以系统退回到 fallback 规则引擎了

你要检查：
- `ollama serve` 是否启动
- 模型是否已 pull
- 后端配置的模型名是否正确

## 14.3 前端能连后端，但播报不更新
这通常说明：
- WebSocket 没连成功
- 或者后端定时任务没工作

你要检查：
- `/ws` 是否连接成功
- 后端日志是否每几秒有一次广播输出

## 14.4 UMG 里嵌网页后显示空白
重点检查：
- Web Browser Widget 是否启用
- 地址是否可被本机访问
- 你的防火墙是否拦截本地端口
- 是否用了 `127.0.0.1` 但 UE 和服务不在同一机器

如果不在同一台机器，必须改成真实 IP。

## 14.5 UE5 看到了播报，但场景没联动
这说明：
- 数据通了
- 但是你还没把“播报等级 -> 场景动作”映射写到蓝图/C++ 里

也就是说：
- 接口打通 ≠ 场景会自动变
- 你还需要自己在 UE5 里写规则映射
